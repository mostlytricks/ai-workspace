01Mission

ai-workspace is a local-only management layer that lets one person run many agent-assisted software projects side by side without anything getting lost between sessions. Scattered terminals forget; chat context evaporates; parallel projects blur together. This workspace is the antidote: a calm root with a fixed storage model, a per-project memory, and a deterministic lifecycle an agent can operate.

north star: one calm root an agent can drive, where no project's context is ever lost
first wedge: a Windows junction-based tier model + per-project handoff docs + lifecycle slash commands
later shape: a portable (Windows / Linux / WSL) workspace standard any coding agent can run
Rules vs why. CLAUDE.md says what to do; this file says why it's worth doing. When a rule feels arbitrary, the justification should be here — and when the mission changes, the rules follow, not the reverse.

02Why This Shape

The workspace earns its discipline with three moves, each aimed at a specific way multi-project work falls apart:

Failure it preventsThe move
"Where do the real files live, and what's active?"Separate storage from views — canonical files in repos/, activity expressed as junction views in tiers.
"What was I doing here last week?"Every project carries its own memory — CLAUDE.md (identity) + CONTEXT.md (rolling now), read at session start.
"Did I run those setup steps right?"Automate the lifecycle — slash commands + helper scripts, so create / promote / move / triage are deterministic, not hand-typed shell.

03Principles

  • Local-first, root is local-only. Never git init at the root or push the workspace as an umbrella repo. Version control lives per project, inside repos/<name>/.
  • Storage vs views. repos/ is the one canonical store; tiers are cheap junction views. Changing a project's tier is a metadata move, never a file copy.
  • The docs are the memory. CLAUDE.md is stable identity (auto-loads); CONTEXT.md is a rolling snapshot of now. git is the changelog — so CONTEXT.md stays small and never becomes a log.
  • One venv per project. No shared or root-level .venv; silent version collisions are the enemy.
  • The lifecycle is agent-operated and deterministic. Prefer slash commands and argv-driven helper scripts over hand-run shell strings — quoting pitfalls (e.g. junction creation) are designed out, not worked around.
  • One index of record. PROJECTS.md is the single source of truth for which tier each project lives in; it's updated on every transition.
  • Right-sized ceremony. Two docs for most projects; the optional four-doc pipeline only for ambitious ones; incubator/ one-shots carry none.

04Architecture Principle

Everything reduces to one idea: real files in repos/, junction views in tiers. Lifecycle commands, triage, and the dashboard all operate on that single model — so a new capability extends the storage/view split rather than inventing a parallel notion of "where a project is."

# canonical storage — the only place real files (.git, .venv, code) live
repos/<project>/
      ^
      |  directory junction  (mklink /J on Windows  ·  symlink on Linux/WSL)
      |     via .claude/scripts/link_project.py — argv-driven, no shell quoting
      |
active/  dormant/  archive/   # views — a project's tier = its activity state
      |
      v
slash commands · /triage · /dashboard · /mission   # all read this one model

(incubator/ is the deliberate exception — real folders, not junctions, because experiments are ephemeral and may never graduate.)

05The Doc System

The workspace runs on documents with distinct, non-overlapping jobs — at the root, and inside each project.

DocumentScopeJob
CLAUDE.mdrootThe rules and invariants an agent must follow. Auto-loads every session.
MISSION.htmlrootThis file — the durable why behind those rules.
HANDBOOK.mdrootHuman-facing workflows, slash-command reference, glossary.
PROJECTS.mdrootThe live index — which project is in which tier, and its current focus.
CLAUDE.md · CONTEXT.mdper projectStable identity + rolling now. The mandatory pair every project carries.
MISSION.html · IMPLEMENTATION_PLAN.mdper projectOptional "four-doc pipeline" for ambitious projects — durable why + phase roadmap (CLAUDE.md §6).

Lifecycle is driven by workspace slash commands — /init-project, /promote, /triage, /dashboard, /mission — defined in .claude/commands/ and backed by helper scripts in .claude/scripts/.

06Non-Goals

  • Do not git init at the root or turn the workspace into an umbrella / monorepo. Projects stay independently versioned.
  • Do not create a shared or root-level .venv.
  • Do not put real project files inside active/, dormant/, or archive/ — those hold junctions only (incubator/ is the one real-folder exception).
  • The workspace does not manage credentials or secrets; that stays with each project and the local toolchain.
  • CONTEXT.md is not a changelog — never duplicate git history into it.
  • Do not grow ceremony where it doesn't pay: small and one-shot projects stay on the light setup.
