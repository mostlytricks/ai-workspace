# Changelog тАФ gravity (the ai-workspace system)

Notable changes to **gravity itself** тАФ the rules in `CLAUDE.md`, the `*.template.*`
stencils, and the `.claude/` commands/scripts. This versions the **methodology**, not
the projects it manages (each project carries its own `CHANGELOG.md`).

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); gravity adheres
to [Semantic Versioning](https://semver.org/spec/v2.0.0.html), read for a *convention system*:

- **major** тАФ a rule projects depend on **breaks**: a renamed/moved convention, a changed
  doc-ownership boundary, a restructure an existing project would now violate. (Example:
  flat `SPEC.<domain>.md` тЖТ `.gravity/<domain>/SPEC.md` was a breaking restructure.)
- **minor** тАФ a new template, command, optional doc, or additive rule. Existing projects stay valid.
- **patch** тАФ clarifications, tighter wording, typo/format fixes; no behavioral change to the rules.

The version lives in the **git tag** (`vX.Y.Z`) and the `VERSION` file тАФ never in prose.
A project records the gravity version it was built on via the `> gravity: vX.Y` line in its
root-`CLAUDE.md` router (seeded from `GRAVITY.template.md`), so drift is detectable.

Ambition-gated (small tools get two docs, no MISSION/PLAN ceremony); ends with a read-back confirmation; never commits. Two modes, one core: **seed** (blank/partial docs) and **re-interview** (only the OPEN items of a stuck project).
- **Feature-scoped interview + growing-project PLAN conventions** тАФ for projects that grow by accreting features rather than marching an arc. **`/interview <project> <feature>`** is the intake ritual: runs `/new-domain`'s *is-it-a-domain* gate (default verdict: *slice* тАФ anti-sprawl), elicits the **use scenario** as `given/when/then`, then writes either a wired new domain or a `PLAN.<slug>.md` slice plus its spine/queue row. **`templates/PLAN.template.md`** (new, the 11th stencil) is the per-domain/per-slice "what/next" doc тАФ Goal ┬╖ **Scenario** (`given/when/then`) ┬╖ Slice ┬╖ Verification ┬╖ `OPEN:` questions ┬╖ Next тАФ now also what `/new-domain` seeds its starter PLAN from. **`IMPLEMENTATION_PLAN.template.md` gains a SLICE-QUEUE VARIANT** (one template, two shapes, like the SPEC integration variant): rolling `now / next / later` lanes with exactly one slice in `now`, replacing fake phases on accretive projects. The maturation pipeline is now explicit end-to-end (CLAUDE.md ┬з6 + SPEC template): a scenario enters as PLAN intent and is **promoted to the SPEC Behavioral Contract only once a named test asserts it** тАФ intent graduates to contract by earning a wall, never by rewording.
- **Golden scenario for `/new-spec`** (`.claude/scenarios/new-spec/`) тАФ guards against the *fabricated wall*: a SPEC whose Gate names a script that doesn't exist, whose `[test:<name>]` points at no test, or which ships template leftovers. The fixture is a minimal but *real* npm project (a `model` domain with `src/model.mjs`, a `lint:model` gate that exits non-zero, two named `node:test` cases тАФ and no SPEC), so an honest SPEC has everything to cite and anything else is fabrication. `check.py scenario` gains an `expect.json` `spec_honesty` flag (spec-honesty FAILs fail the scenario); `selftest` now validates **every** scenario's golden fixture (consistency + spec honesty), so a rotted fixture can't make its scenario prove nothing. Replayed end-to-end at authoring: honest run **PASSED**; a copy with one fabricated `[test:imaginary-coverage]` tag correctly **FAILED** (`TAG_DEAD`).

## [1.2.0] - 2026-07-02

### Added
- **Golden-scenario harness for gravity's own commands** (`.claude/scenarios/`) тАФ gravity now tests the commands that *claim* to wire things up. A scenario is a triple: `(command, golden-input fixture, deterministic structural assertions)`. The agent step (running the command) stays manual; the assertion is a script. First scenario: **`/new-domain`** тАФ copies a clean one-domain fixture, the agent adds a second domain, and the checker proves it's wired into all four registry indexes with nothing orphaned (catches the half-wired/orphaned-domain bug).
- **`check.py` тАФ structural-invariant checker** (`.claude/scenarios/check.py`) тАФ `check_gravity_consistency(project_dir)` reads the four registry owners (Doc Map, router table, MISSION "system in N domains" row, `IMPLEMENTATION_PLAN.md` status spine) and reports `UNDERWIRED` domains and `ORPHAN_ROUTE` rows via heuristic slug-match. CLI: `consistency` (one project), `scenario` (assert a golden-scenario's postconditions), `selftest` (proves the checker against a good + deliberately-broken fixture).
- **`/triage` runs `check.py`** тАФ the `.gravity/` registry-drift flag is now mechanical instead of by-hand: triage shells out to `check.py consistency` per `.gravity/` project, so triage and the `/new-domain` scenario share one checker and agree by construction.
- **Project aliases тАФ `resolve_project.py`** (`.claude/scripts/`) тАФ a short token resolves to a canonical project with **no central registry** (PROJECTS.md stays the workspace index, not an alias table). The alias is a per-project identity fact, so a project **declares its own** via a `> alias: <slug>` line in its root `CLAUDE.md` (seeded from `CLAUDE.template.md`, next to the `> gravity:` stamp) тАФ authoritative and owned per project. Resolution order: exact name тЖТ **declared alias** тЖТ unique acronym as a convenience (`amos` = architecture-memory-os, short тЙд2-char tail segments like `os` kept whole) тЖТ unique substring (`compass`), refusing to guess when ambiguous. So a project gets a free derived alias immediately and can *claim* an intentional one. `check.py --project <alias>` resolves them; other project-taking commands can adopt the same helper. First declared alias: **`amos`** (architecture-memory-os).
