01What gravity is

Gravity is the operating manual + toolset of this workspace: a small set of documents with strict ownership rules, a directory convention (.gravity/), mechanical checkers, and slash commands — which together let AI coding agents work on many projects without losing the thread. The name is the core metaphor: every fact has exactly one home it gravitates to, and every project has enough mass (mission, rules, plan) that agents orbit it correctly instead of drifting.

It is not a framework you install. It's a discipline the workspace enforces: templates you copy, indexes that must stay wired, enforcement tags that must be honest, and checkers (check.py, /triage) that catch the drift mechanically.

02The problem it solves

Agent sessions are powerful and amnesiac. Between sessions, machines, and model generations, three things evaporate:

  • State — where was I? what's next? (the resume problem)
  • Rules — what must never break here? which conventions are real walls vs. taste? (the safety problem)
  • Intent — why does this exist? what are we deliberately not doing? (the direction problem)

No model, however capable, can re-derive choices — non-goals, locked decisions, boundary contracts. They exist only in your head or in a chat that scrolled away. Gravity is a persistence layer for that judgment: each concern gets one document, at one rate of change, loaded at the right moment.

03The workspace

One root, four tiers, one index. Projects live in repos/ (storage); tier folders hold junctions (views) — moving a project between tiers is instant metadata, never a file copy.

ai-workspace/
├── CLAUDE.md            # the operating manual — auto-loads, the one router
├── PROJECTS.md          # live index: every project, tier, focus, gravity adoption
├── VERSION · CHANGELOG.md  # gravity itself is versioned (see §10)
├── docs/ · templates/ · .claude/   # read-docs · stencils · commands + scripts
├── repos/               # CANONICAL storage — real project files, one folder per project
├── active/ · dormant/ · archive/   # junction views by lifecycle tier
└── incubator/           # cheap experiments, no ceremony

The root repo tracks only the skeleton (deny-all .gitignore, whitelist the meta files) — each project is its own independent git repo. A project is the unit of attention, not the unit of git: a project usually holds one repo, but a multi-service system can be a hub project — a docs-only repo with the service clones nested inside, git-denied (§08).

04The doc model — four clocks

Documents are separated by how fast they change, because mixing rates is what makes docs rot. Two auto-load from every project root; the rest are opt-in and recognized only when present.

DocQuestionChanges
MISSION.htmlwhy — north star, principles, non-goalsrarely
CLAUDE.md auto-loadshow — identity, stack, run/test, routingon refactors
IMPLEMENTATION_PLAN.mdwhat/next — phases or slice queue, tracks, locked decisionsper phase/slice
CONTEXT.md auto-loadsnow — current state + the single next stepevery session
.gravity/<domain>/SPEC.mdthe agent contract for changing one domainwhen rules change
.gravity/<domain>/PLAN.*.mdthe intent of one change — goal, scenario, slice, verificationper slice
WALKTHROUGH.mdproof — what shipped + the evidence it worksappend-only
One concern, one home. Every fact has exactly one owner-doc; every other doc links to the owner instead of restating it. The same fact written twice eventually drifts into two different facts.

05.gravity/ and the protocol card

When a project outgrows its flat root, the heavy docs move into a .gravity/ directory, grouped by subject domain, not doc-type. The directory is the domain registry — there is no registry file; existence, routing, why, and status live split across four rate-appropriate owners (the folders, the CLAUDE.md Doc Map, the MISSION rows, the PLAN spine), and check.py consistency catches any of the four going stale.

<project>/
  CLAUDE.md · CONTEXT.md · README.md      # stay at root — the router, now, the guide
  .gravity/
    GRAVITY.md                            # the protocol card (versioned copy — below)
    MISSION.html · ARCHITECTURE.html · IMPLEMENTATION_PLAN.md
    <domain>/    ARCHITECTURE.html · SPEC.md · PLAN.*.md
    integration/ SPEC.md   # the contracts BETWEEN services/domains + the change order

The protocol card makes each repo self-describing. Projects are independent repos — an agent cloning one without the workspace would find SPECs and PLANs with no explanation. So every .gravity/ ships GRAVITY.md: a verbatim, version-stamped copy of the project-side protocol (doc kinds, navigation discipline, SPEC anatomy). Never hand-edited; upgrading gravity means re-copying it, and /triage flags a stale card.

06SPECs — walls, not wishes

A domain's SPEC.md is the sheet an agent loads before changing that domain. It is two halves at once:

Generative build from

A Minimal Shape + a short Generate loop — the template a correct new unit is instantiated from.

Limiting fenced by

Rules where every rule carries an enforcement tag naming the wall that catches a violation — or admits there is none.

TagMeaning
[lint] / [type] / [test:name]a real wall — a named check fails if the rule breaks
[review] / [—]human judgment only — honest about having no wall
The honesty rule. Never tag a rule with a wall you haven't verified exists — under-claim to [review] when unsure. check.py spec re-verifies every Gate and tag against the repo's reality, so a renamed test can't silently turn a wall into a lie.

Behavior matures by graduation: a scenario enters a PLAN as given/when/then intent, and is promoted into the SPEC's Behavioral Contract only once a named test asserts it. Intent earns a wall — it is never reworded into one.

07Managing direction

Gravity refuses to store direction in its abstract form ("queries should be compact…") — an unowned wish rots. A direction must earn one of four homes, each with its own management mechanism:

the direction pipeline
MISSION principle  →  Track              →  domain PLAN slices  →  SPEC rules
forever; wins        the cross-domain       the scheduled work      the residue: applies to
arguments; /triage   arc — an index of      (goal · scenario ·      every future change
flags drift          ≤3 active intents      slice · verification)   automatically, with walls

A Track (in IMPLEMENTATION_PLAN.md) is the direction axis the spine can't show: it holds no slices itself, points up to its MISSION principle and down to the domain slices carrying it — and retires when its residue is walls: the direction lives on as enforcement-tagged SPEC rules, needing no further management.

08Brownfield — archaeology before authorship

On a mature system you didn't write, greenfield's docs-then-code order inverts, and the integration domain comes first — the seams are exactly what you don't know. /excavate scans the code (never the DB), joins the evidence on three keys (table · path+method · base URL), and writes a Boundary Map where every row cites its source file; a seam that can't be traced stays an honest OPEN: line, never a guess.

Two escalations for hard cases:

  • The DB evidence pack — when the code carries no readable queries (dynamic SQL, stored procedures), a DBA exports metadata-only CSVs offline (columns+comments, FK graph, grants, activity; never row data). Partial packs are fine by design; the FK graph + name prefixes + shared grants cluster tables into candidate vertical business domains.
  • The hub project — when the system spans many service repos, a docs-only repo holds the cross-service gravity, with the service clones nested under services/ and git-denied — the workspace's own deny-all pattern, one level down.

Gravity tests its own commands: .claude/scenarios/ holds golden fixtures with deterministic assertions, and check.py selftest proves the checkers themselves.

10Gravity is versioned

The methodology itself follows SemVer — major = a rule projects depend on breaks. The version lives in the VERSION file + a git tag, never in prose. Each project records what it adopted in two stamps: the > gravity: vX.Y line in its router and the protocol card's stamp — so stale adoptions are visible (adoption table in PROJECTS.md, chips on the dashboard, 📡 in /triage) and repairable (/sync-gravity).

11The philosophy

  • Weaker-agent walls. Wherever judgment can fail, build a mechanical wall instead: argv-driven scripts, verbatim copies, versions that are read, never invented. The system is designed so a less capable agent can't fabricate enforcement — and a more capable one doesn't have to be trusted.
  • Honesty over completeness. An unknown is an OPEN: line, not a plausible guess. An unenforced rule is [review], not a fake wall. A partial evidence pack is a valid state.
  • Ceremony must pay. Two docs are the minimum and often the maximum. Everything else is recognized only when present, gated by ambition, and deleted when it stops earning its keep.
  • Intent is the scarce resource. Models get stronger and code gets cheaper; your decisions, non-goals, and boundaries do not regenerate. Gravity exists to make those durable — and to apply them automatically to work you'll never personally watch.
