# Gravity — 시스템 개요 및 커버리지

> 개발 실무자 관점에서 전체 시스템을 한 번에 읽는 문서.
> 기준 버전: **gravity v2.5.1 + unreleased** (2026-07). 영문판: `OVERVIEW.md`.
> 상세 문서: `CLAUDE.md`(에이전트 계약) · `HANDBOOK.md`(워크플로) · `.claude/scenarios/README.md`(체커 레퍼런스).

---

## 1. Gravity란 무엇인가

Gravity는 여러 소프트웨어 프로젝트를 AI 코딩 에이전트와 함께 — 세션, 머신, 서로 다른 모델(Claude, Codex 등)을 넘나들며 — 운영하기 위한 **방법론이자 기계적 도구 체계**다. 공략하는 실패 모드는 단 하나다.

> **세션 간 지식 손실.** 에이전트는 기억이 없고, 사람은 잊으며, 채팅은 증발한다. 살아남아야 하는 모든 것 — 의도, 규칙, 상태, 출처, 증명 — 은 지정된 소유자가 있는 파일에 존재해야 하며, 그렇지 않으면 사라진다.

시스템의 모든 구성이 여기서 파생된다. 문서는 영속성 계층이고, 커맨드는 문서를 정직하게 유지하는 의례(ritual)이며, 체커는 거짓말하는 문서를 잡아내는 기계적 방벽(wall)이다. Gravity는 의도적으로 프로젝트 관리 도구가 **아니다** — §10 참고.

## 2. 설계 원칙

1. **하나의 관심사, 하나의 집.** 모든 사실은 정확히 하나의 소유 문서를 가지며, 다른 문서는 링크한다. 같은 사실을 두 곳에 쓰면 결국 서로 다른 두 "진실"로 표류한다.
2. **문서마다 변화 속도가 다르다.** *왜*(드물게) · *어떻게*(리팩터링 시) · *다음 할 일*(슬라이스마다) · *지금*(세션마다). 변화 속도가 섞인 문서는 가장 빠른 내용의 속도로 썩는다.
3. **사실은 스크립트가, 판단은 에이전트가.** 결정적 스크립트가 스캔·검증·빌드하고, 에이전트가 해석하고 결정한다. 이 경계는 절대 흐리지 않는다.
4. **산문의 약속보다 기계적 방벽.** "에이전트가 제대로 했는가, 아니면 했다고 말만 했는가?" 에이전트가 할 수 있는 모든 주장 — 인덱스 배선 완료, 스펙 집행 중, 롤백 증명, 시트 라우팅 완료 — 에는 그 거짓을 잡아낼 체커가 존재한다.
5. **지어내기보다 증거.** 모르는 것은 `OPEN: awaiting …`으로 보이게 남기고, 그럴듯하게 채우지 않는다. 재현 없으면 수정 없음. 증거 없으면 스키마 변경 없음.
6. **의례는 선택 사항(opt-in).** 필수 문서는 둘뿐이며, 나머지는 전부 *있으면 인식하되, 강제하지 않는다*. 스스로 값을 못 하는 의례는 규율이 아니라 표류다.
7. **레시피는 줄고, 방벽은 자란다.** 모델이 좋아질수록 단계별 절차 텍스트는 짧아지고 체커 코드베이스는 커져야 한다. (v2.1 이후 궤적이 이를 증명: 산문 → 스크립트.)

## 3. 워크스페이스 계층

```
ai-workspace/            # 스켈레톤 저장소 (deny-all .gitignore — 화이트리스트가 곧 이식 가능한 스켈레톤)
├── CLAUDE.md            # 에이전트 운영 매뉴얼 — 자동 로드; 단어 예산 방벽(MANUAL_BLOAT) 적용
├── docs/                # 사람이 읽는 문서 (본 문서, HANDBOOK, INTRO, MISSION, intake 시트)
├── templates/           # 16개 스텐실 — 프로젝트로 복사될 뿐, 자동 로드되지 않음
├── .claude/commands/    # 20개 슬래시 커맨드 (lazy: 호출 시에만 로드)
├── .claude/scenarios/   # 검증 하네스 (check.py + 골든 픽스처)
├── repos/               # 프로젝트 정본 저장소 — 각각 독립된 git 저장소
└── active/ stable/ dormant/ archive/   # 티어 폴더: 정션(junction)만, 실제 파일 금지
```

**라이프사이클**은 *작업 중 · 잘 돌아감 · 중단됨 · 종료*로 읽는다. `/init-project` → `active/`; `/ship` → `stable/`(Next Step이 *재활성화 트리거*가 되고, 신선도 검사에서 면제 — 조용함은 성공이다); 일시 중단 → `dormant/`(*재개 차단 요인*을 반드시 명시); `/retire` → 보관 또는 삭제. 티어 이동은 정션 `mv` 한 번 — 즉시 끝나고 git에 투명하다. `PROJECTS.md`는 라이브 인덱스이며 **로컬 전용**이다(비공개 프로젝트명을 담으므로 스켈레톤에는 새니타이즈된 샘플만 포함).

## 4. 프로젝트별 문서 계층

**2문서 최소 구성(필수):**

| 문서 | 질문 | 주기 |
|---|---|---|
| `CLAUDE.md` | *어떻게* — 정체성, 스택, 실행/테스트, 컨벤션, Doc-Map 라우터 | 리팩터링 시 |
| `CONTEXT.md` | *지금* — Completed(최근 1–2세션) · Current State · 단 하나의 Next Step | 매 세션; 정기 정리(~80줄) |

**4문서 파이프라인(실제 아크가 있는 프로젝트):** `MISSION.html`(*왜* — 북극성, 원칙, non-goal; `/triage`가 표류를 대조하는 기준)과 `IMPLEMENTATION_PLAN.md`(*다음 할 일* — 도메인 상태 스파인, `now`에 정확히 하나의 슬라이스만 두는 슬라이스 큐, 확정된 결정, 게이트)를 추가.

**`.gravity/`(플랫 루트를 벗어난 프로젝트):** 자동 로드되는 두 파일을 제외한 모든 문서가 닷 디렉터리 아래로 들어가며, **문서 종류가 아니라 주제 도메인별로** 묶인다. 각 도메인 폴더는 필요한 종류만 갖는다.

| 종류 | 역할 |
|---|---|
| `SPEC.md` | **에이전트 계약**: 생성 기준인 *Minimal Shape* + **모든 규칙에 집행 태그**(`[lint]`/`[type]`/`[test:name]`/`[review]`/`[—]`)가 붙은 *Rules* — 어떤 규칙이 실제 방벽이고 어떤 것이 리뷰어 판단인지 계약이 거짓말하지 않는다. 행위 중심 도메인은 `given/when/then` **Behavioral Contract**를 추가하며 각 줄은 명명된 테스트에 결속된다. |
| `ARCHITECTURE.html` | 파일 맵을 넘어서는 경우의, 사람을 위한 심층 문서(근거·설계 이유) |
| `PLAN.*.md` | 슬라이스 하나의 의도: Goal · Scenario(`given/when/then`) · Slice · Verification. **버그는 *현재-거짓* 시나리오(재현)로 여기에 진입**하고, 수정은 그 시나리오를 SPEC으로 승격시키는 회귀 테스트를 남겨야 한다. |
| `given/` + `MANIFEST.md` | 전달받은 도메인 지식(§5) |

**도메인 레지스트리 파일은 없다** — 디렉터리 자체가 레지스트리이며, 변화 속도가 다른 네 소유자로 분산된다: 존재(폴더) · 라우팅(루트 CLAUDE.md의 Doc Map + read-first 테이블) · 왜(MISSION 행) · 상태(PLAN 스파인). `check.py consistency`가 어느 소유자에서든 누락된 도메인을 FAIL로 잡는다. **프로토콜 카드**(`.gravity/GRAVITY.md`, 템플릿 원문 복사본 + 버전 스탬프)는 워크스페이스 없이 저장소만 열어도 스스로를 설명하게 만든다.

**선택 문서(있으면 인식):** `DESIGN.md`(앱 시각 계약) · `RUNBOOK.md`(운영 — "새벽 2시 테스트") · `docs/walkthroughs/`(날짜별 append-only *사후* 기록: 무엇이 출하됐고 그 증명) · `AGENTS.md`(Codex 심).

## 5. 증거 계층 — 들어오는 두 개의 문

두 문 모두 **전달받은** 자료(결정된 진실이 아님)를 담고, 쌓이지 않고 **라우팅되어 나간다**.

| | Intake (신고된 주장) | Given (전달받은 지식) |
|---|---|---|
| 입력 | 사용자/QA의 버그·이슈 신고 | 데이터 사전, 업무 규칙, 벤더 문서, 조직 컨텍스트 |
| 착지 | `docs/intake/<날짜>.md` — 배치당 시트 하나, 신고문은 **원문 그대로** | `.gravity/inbox/` — git-ignored 드롭 존(트리아지가 프라이버시 등급을 정하기 전에는 아무것도 커밋될 수 없음) |
| 의례 | `/intake`: 항목당 6개 필수 사실(질문으로 끌어내거나 `OPEN:`), 트리아지 3문(진짜인가? → 어느 도메인? → 버그/기능/문서 표류?), **근본 원인**으로 중복 제거 | `/given`: 확인받은 라우팅 테이블 하나 → `.gravity/<도메인>/given/`으로 이동 + 출처 행(source · received · version · **fidelity** · privacy) |
| 강행 규칙 | **재현 없으면 슬라이스 없음**; 버그는 절대 도메인이 아니다 | **채석장이지 계약이 아니다** — 인용하되 재서술 금지; 분쟁은 렌더링이 아니라 `raw/` 원본 기준으로 해결 |
| 방벽 | `check.py intake` | `check.py given` |

**유지보수 루프**는 문에서 문까지 이어진다: 신고 → intake 시트 → 버그-intake 슬라이스 PLAN(현재-거짓 시나리오) → `/patch-slice`(기계적 방벽: 앵커, 스냅숏, bare-gated 검증 N=3 → 실패 시 4중 증명 롤백 강제) → 회귀 테스트가 시나리오를 SPEC으로 승격. 버그 수정은 정직한 계약 조항의 가장 빠른 공급원이다.

## 6. 커맨드 표면 (20개)

커맨드는 **lazy** — 호출 시에만 로드되며, 상시 컨텍스트 비용은 CLAUDE.md 하나뿐(단어 예산 방벽 적용). 파괴적 커맨드(`/retire`, `/cut-release`, `/ship`)는 `disable-model-invocation` — 사람만 호출 가능. 모델이 스스로 선택할 수 있는 커맨드에는 **트리거 펜스**("언제 쓰면 안 되는가")가 붙는다.

| 그룹 | 커맨드 |
|---|---|
| 라이프사이클 | `/init-project` · `/ship` · `/retire` |
| 문서 시스템 | `/adopt-gravity` · `/sync-gravity` · `/new-domain` · `/new-spec` · `/excavate`(브라운필드: 저작 전에 고고학 — 출처 명기된 Boundary Map, 모르는 것은 `OPEN:` 유지) |
| 증거·도출 | `/interview`(머릿속 → 문서, 스트로맨 우선) · `/intake`(신고) · `/given`(지식) · `/mission`(재정렬, 읽기 전용) |
| 유지보수 | `/patch-slice`(스크립트 방벽이 감싼 패치 루프) |
| 릴리스 | `/cut-release`(Change Order: `[Unreleased]` 증거로 확인받은 범프, 게이트 그린 필수, 푸시 직전 정지) |
| 조망·뷰 | `/triage`(주간 표류 리포트; 기계적 스캔) · `/dashboard` / `/open-dashboard` · `/cosmos`(프로젝트를 항성계로 렌더링 — 인덱스에서 라이브 스캔) · `/open-mission` / `/open-architecture` |

## 7. 검증 하네스와 커버리지

`check.py`(약 1,200줄) + `scan_workspace.py` + `patch_slice.py`가 방벽이다. 체커 하나, 호출자 여럿 — 같은 함수가 골든 픽스처(시나리오)와 라이브 저장소(`/triage`) 양쪽에서 돈다.

| 서브커맨드 | 지키는 것 | 주요 발견 |
|---|---|---|
| `consistency` | 4-소유자 도메인 레지스트리가 거짓말하지 않음 | `UNDERWIRED`(FAIL) · `PROTOCOL_STALE` |
| `spec` | 집행 태그가 시간이 지나도 참 | `GATE_DEAD` · `TAG_DEAD` · `SPEC_UNFILLED`(FAIL) + 도메인별 태그 센서스(방벽 vs `[review]`) |
| `workspace` | 티어/인덱스/도입 표류; 매뉴얼 자체의 크기 | `MULTI_TIER`(FAIL) · `MANUAL_BLOAT` |
| `intake` | 시트 정직성 | `INTAKE_UNROUTED`(거짓말하는 ✓ 시트에서 FAIL) · `INTAKE_DEAD_ROUTE` |
| `given` | 인박스 라우팅 완료, 출처 존재, 매니페스트 진실성 | `GIVEN_GHOST_ROW`(FAIL) · `INBOX_UNROUTED` |
| `scenario` | 골든 픽스처 위에서 커맨드의 구조적 사후 조건 | `expect.json` 기준 |
| `selftest` | **검증자 자신**: 모든 픽스처가 깨끗하고, 모든 체커가 심어 놓은 표류를 잡으며, `patch_slice.py`를 그린·레드 양 분기로 끝까지 구동(강제 롤백이 git-ignored 상태를 바이트 단위로 복원) | `SELFTEST PASSED/FAILED` |

**커버리지 맵(정직하게):** 골든 시나리오가 있는 커맨드 — `/new-domain`, `/new-spec`, `/excavate`, `/ship`, `/patch-slice`; 인코드 selftest 하프가 커버 — `spec`, `workspace`, `intake`, `given`, patch-loop 스크립트. **아직 시나리오 미커버:** `/init-project`, `/adopt-gravity`, `/sync-gravity`, `/cut-release`, `/interview`, `/retire`, 뷰 커맨드들. 에이전트의 *판단* 단계는 설계상 테스트 불가 — 구조적 사후 조건만 단언한다. **CI는 없다**: 모든 방벽은 온디맨드로 실행(§10 참고); 하네스를 수정한 뒤에는 selftest가 게이트다.

## 8. 버저닝과 업그레이드

Gravity는 **컨벤션 시스템으로서** SemVer를 따른다: major = 프로젝트가 의존하는 규칙 파괴 · minor = 새 템플릿/커맨드/추가 규칙 · patch = 문구 수정. 버전은 `VERSION` 파일 + git 태그에 존재하며 산문에는 절대 없다. 각 프로젝트는 도입 버전을 기록하고(라우터의 `> gravity: vX.Y` 스탬프 + 프로토콜 카드 스탬프), `/sync-gravity`가 기계적 부분(카드 재복사 + 스탬프)을 적용하되 판단이 필요한 델타는 인용 체크리스트로 **보고만 하고 절대 자동 마이그레이션하지 않는다**. `/cut-release`는 gravity 자신과 프로젝트에 동일한 Change Order를 적용한다.

## 9. 피드백 루프 (실제 사례 일주일)

현장 리포트 → `docs/intake/` 시트(원문, 6개 사실) → 수정 → **회귀 방벽** → 종종 새 규칙. 2026년 7월의 실제 사례: 코딩 에이전트가 일반 코드 분석 작업에 `/intake`를 과잉 선택 → 트리거 펜스 + 펜스-출생시-장착 규칙(v2.5.1); Linux에서 selftest가 레드(`__pycache__` 오염) → 수정 + intake 항목 I1로 기록; 매뉴얼이 컨텍스트 예산을 초과 → §7을 포인터 테이블로 압축 + `MANUAL_BLOAT` 방벽 신설 — 그리고 그 방벽이 바로 *다음* 기능의 초안에서 정확히 발화했다. 시스템이 자기 자신의 성장을 다스린다.

## 10. 범위 펜스 — Gravity가 아닌 것

- **스프린트/보드/추정/지표가 아니고, CI도 아니다** — gravity는 *세션 간 지식 손실*이 실패 모드인 라이프사이클 구간만 다룬다. 파이프라인과 프로세스 관리는 다른 도구의 일이다.
- **엄브렐러 저장소가 아니다** — 워크스페이스는 스켈레톤만 추적하고, 모든 프로젝트는 자체 리모트를 가진 독립 저장소다.
- **능력 보조 장치가 아니다** — 방벽은 프론티어 모델에도 여전한 실패 모드(세션 기억상실, 관성 오류, 그럴듯하지만 틀린 주장, 의례 과잉 적용)를 겨냥한다. 줄어들 것으로 기대되는 쪽은 레시피다.
- **강제 의례가 아니다** — CLI 스크립트 하나는 영원히 문서 두 개로 행복하게 산다.
